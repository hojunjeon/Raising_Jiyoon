## 게임을 키우는 느낌을 어떻게 만들 것인가

초기 설계는 단순했다. **플레이어가 여러 NPC와 1:1로 채팅하는 구조**. 입과식에 가서 선배를 만나면 "안녕하세요?"라고 먼저 말을 걸고, 선배가 응답하는 식이다.

그런데 이 방식엔 문제가 있었다. 플레이어가 **항상 먼저 말을 걸어야** 한다. 게임의 핵심이 "김지윤을 키운다"인데, 플레이어가 NPC를 대하는 주체가 되어버리니 실제로는 "NPC들과 상호작용한다" 느낌이 났다. 수동적이었다.

**씬 기반 구조로 전환하면 이게 달라진다.**

입과식 씬이 시작되면, 플레이어는 그 상황에서 **김지윤이라면 뭐라고 할까**를 고민한다. 선배가 먼저 인사를 건넨다. 플레이어는 김지윤의 답변을 입력한다. 그러면 선배가 그 답변에 반응한다.

이렇게 되면 게임은 완전히 달라진다. 플레이어가 김지윤의 대리인이 되어 상황을 읽고 판단하는 경험이 생긴다. "내가 이렇게 말하면 이 사람이 어떻게 생각할까?"를 고민하게 되고, 그 선택이 김지윤의 능력치에 반영된다.

## 무엇을 바꿨는가

변경이 생각보다 광범위했다. 변경 전후를 비교해보면:

| 항목 | 변경 전 (NPC 1:1) | 변경 후 (씬 기반) |
|------|---|---|
| **URL** | `/chat/:characterId` | `/scene/:sceneId` |
| **메인화면** | 캐릭터 카드 목록 | 씬 카드 목록 (입과식 / 알고리즘 / 팀프로젝트) |
| **플레이어 입력** | "내가" 하는 말 | "김지윤이" 하는 말 |
| **API 요청** | `{ character_id, message }` | `{ scene_id, protagonist_line }` |
| **API 응답** | `{ reply, stat_delta, choices }` | `{ speaker, reply, stat_delta, choices }` |
| **UI 표시** | 말풍선에 이름 없음 | NPC 이름 레이블 표시 |

한 두 곳을 수정하는 게 아니라:

**프론트엔드 (6개 파일)**
- App.jsx — 라우트 구조 변경
- gameApi.js — API 요청 함수 재설계
- ChatScreen.jsx — 씬 기반 채팅 로직
- MessageBubble.jsx — speaker 필드 처리
- MainScreen.jsx — 씬 목록 렌더링
- global.css — 새 스타일 추가

**백엔드 (3개 파일)**
- chat.py — API 엔드포인트 로직 변경
- mock.py — 씬별 응답 + speaker 필드

**시나리오 (3개 파일)**
- scene_1.md, scene_2.md, scene_3.md — 대사, 선택지, 능력치 구조

총 12개 파일을 동시에 건드려야 했다.

## Claude Code Plan Mode의 진가

이렇게 큰 변경을 일관성 있게 할 수 있었던 핵심은 **Claude Code의 Plan Mode**를 제대로 활용한 것이다.

### 계획 단계에서 범위를 명시했다

계획 파일에 다음을 명확히 기록했다:

**변경하는 것:**
- API 스키마: `character_id + message` → `scene_id + protagonist_line`
- 프론트엔드 라우트: 캐릭터 기반 → 씬 기반
- 응답 구조: speaker 필드 추가

**변경하지 않는 것:**
- 능력치 시스템 (stat_delta 구조 유지)
- 선택지 메커니즘
- StatToast 컴포넌트
- 인증 로직

이 명시가 중요했다. 에이전트에게 "다른 건 건드리지 마"라는 신호를 미리 주면, 작업 중에도 범위 침투(scope creep)가 일어나지 않는다. "혹시 능력치 시스템도 개선하면 좋을까?" 같은 불필요한 확장 논의 없이 핵심만 바꿀 수 있었다.

### 코드 수준의 설계를 계획에 담았다

계획 파일엔 단순히 "라우트 변경"이 아니라:

```javascript
// 변경 전
async function sendMessage(userId, characterId, message) {
  const response = await api.post('/chat', {
    character_id: characterId,
    message: message
  });
}

// 변경 후
async function sendMessage(userId, sceneId, protagonistLine) {
  const response = await api.post('/chat', {
    scene_id: sceneId,
    protagonist_line: protagonistLine
  });
}
```

이런 식으로 구체적인 코드 변경을 보여줬다. 덕분에 에이전트가 계획을 보고 바로 실행할 수 있었다. 처음부터 "이렇게 해야겠다"는 합의가 있으니, 실행 중 방향 수정이 거의 없었다.

## Speaker 필드를 왜 추가했나

한 씬에 여러 NPC가 등장할 수 있다. 예를 들어 입과식 씬에서 선배 A, 선배 B, 조교가 모두 나올 수 있다.

초기에는 "응답 텍스트만 있으면 되지 않을까?"라고 생각했는데, 막상 구현해보니:

1. **UI 단에서 누가 말하는지 알아야** 말풍선에 이름 레이블을 붙일 수 있다
2. **LLM에게도 컨텍스트가 필요하다** — "이 씬은 선배 A의 관점에서 응답하라"는 정보가 있어야 일관성 있는 응답이 나온다

그래서 API 응답에 speaker 필드를 추가했다:

```json
{
  "speaker": "선배_A",
  "reply": "반갑습니다, 새내기!",
  "stat_delta": {
    "charisma": 5,
    "confidence": 3
  },
  "choices": [...]
}
```

## 남은 과제

계획 단계에서 "변경하지 않는 것"을 명시한 덕에, 지금 남은 이슈는 명확하다:

- **Mock LLM 상태**: 현재 시나리오 파일에 하드코딩된 응답만 있다. 실제 AI 응답은 아직
- **Node.js 미설치**: 프론트엔드 통합 테스트를 아직 못했다
- **HuggingFace 연동**: 다음 작업(llm_integration.md)에서 처리할 예정

이들이 "남은 것"이지 "실패한 것"이 아니라는 게 중요하다. 범위를 명확히 했으니까, 다음 세션에서 바로 이어받을 수 있다.

## 마무리

대규모 리팩토링을 할 때, 코드를 먼저 건드리고 나중에 정리하는 방식도 있다. 하지만 15개 파일을 일관성 있게 바꾸려면 **먼저 계획을 세우고, 그 계획을 명확히 한 뒤에 실행하는 게** 훨씬 효율적이다.

Claude Code Plan Mode의 진가는 여기에 있다. 에이전트도, 사람도 같은 맥락에서 움직일 수 있게 한다.
