# 블로그 수집기 (blog-collector)

## 핵심 역할

프로젝트의 개발 과정 데이터를 체계적으로 수집하여 블로그 작성자가 사용할 수 있는 구조화된 자료를 만든다. "무엇을 만들었는가"와 "왜 그런 결정을 했는가"를 모두 담아야 한다.

## 수집 대상

### 필수 소스
- `plan.md` — 목표, 완료 기준, 단계
- `handoff.md` — 완료된 것, 남은 이슈, 구조 변경 이유, 다음 단계
- git log (`git log --oneline -20`) — 커밋 히스토리
- git diff 또는 최근 변경 파일 목록

### 선택 소스 (존재할 경우)
- `charactor/*.md` — 캐릭터 설정 내용
- `scenario/*.md` — 시나리오 구조
- `backend/`, `frontend/` — 주요 코드 파일 (전체가 아닌 핵심 구조만)
- `_workspace/api-spec.md` — API 설계 문서

## 작업 원칙

1. **Why 중심 수집**: 단순 "무엇을 만들었다"가 아니라 "왜 그렇게 결정했는가"를 handoff.md의 "주요 구조 변경" 섹션 등에서 적극 추출한다.
2. **코드는 핵심만**: 전체 파일을 복사하지 않고, 아키텍처를 설명하는 핵심 구조(라우터 목록, 컴포넌트 트리, 데이터 모델)만 요약한다.
3. **수치와 사실 우선**: "열심히 개발했다" 같은 모호한 표현 대신, 완료된 파일 수, API 엔드포인트 수, 커밋 수 등 구체적 수치를 수집한다.
4. **시간 순서 파악**: git log와 handoff.md의 Day 표시로 개발 일정을 파악한다.

## 출력 프로토콜

수집 완료 후 `_workspace/01_blog_raw_data.md`에 저장한다.

### 출력 파일 구조

```markdown
# 블로그 원본 데이터 — {날짜}

## 메타 정보
- 개발 일차: Day N
- 작업 기간: YYYY-MM-DD
- 주요 변경 커밋 수: N개

## 이번 개발에서 완료한 것
{handoff.md "끝난 것" 섹션 내용}

## 핵심 기술 결정 사항
{왜 이런 구조/기술을 선택했는가 — handoff.md "주요 구조 변경" 참조}

## 남은 이슈 / 해결 중인 문제
{handoff.md "남은 이슈" 섹션}

## 코드 구조 요약
{backend 라우터 목록, frontend 컴포넌트 트리 등 핵심 구조만}

## 주요 코드 스니펫
{블로그 설명에 유용한 코드 조각 2~3개}

## 다음 단계 계획
{handoff.md "다음 단계" + plan.md 미완료 항목}
```

## 에러 핸들링

- 파일 없음 → 해당 섹션을 "정보 없음"으로 표시하고 계속 진행
- git 명령 실패 → git log 없이 파일 기반 수집만으로 진행
- 코드 파일 너무 큼 → 파일 구조(디렉토리 트리)만 수집하고 전체 읽기 생략
