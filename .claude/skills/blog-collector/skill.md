# 블로그 데이터 수집 스킬

김지윤 키우기 프로젝트의 개발 과정 데이터를 체계적으로 수집한다.

## 수집 순서

### Step 1: 작업 디렉토리 확인
```
프로젝트 루트: C:\Users\SSAFY\Desktop\Chat_Game
_workspace/ 폴더가 없으면 생성
```

### Step 2: 핵심 문서 읽기

다음 순서로 읽는다:

1. `devlog/*.md` — **가장 먼저 읽는다.** 작업별 개발 기록. "왜"와 "막혔던 점"이 담겨 있다. `_template.md`는 제외하고 읽는다.
2. `handoff.md` — 세션 인수인계 상태. devlog에 없는 이슈/다음 단계 보완용.
3. `plan.md` — 목표, 완료 기준 체크 상태

### Step 3: Git 히스토리 수집

```bash
git log --oneline -15
git diff HEAD~3 HEAD --stat   # 최근 3커밋 변경 파일 목록
```

오류 발생 시 생략하고 계속 진행.

### Step 4: 코드 구조 요약

**전체 파일을 읽지 않는다.** 구조를 파악할 수 있는 최소한만 수집:

- `backend/` 존재 시: `main.py` 상단 (라우터 include 부분만), `routers/` 파일 목록
- `frontend/src/` 존재 시: 컴포넌트 디렉토리 구조 (`ls` 또는 Glob)
- `charactor/` 존재 시: 파일 목록 + 각 파일 상단 10줄
- `scenario/` 존재 시: 파일 목록

### Step 5: 핵심 코드 스니펫 선택

블로그 설명에 유용한 코드 조각을 2~3개 선택한다. 기준:
- 이번 작업에서 새로 작성했거나 크게 변경된 코드
- 아키텍처 결정을 잘 보여주는 코드
- 독자가 "아 이렇게 구현했구나" 할 수 있는 짧고 명확한 코드 (20줄 이내)

## 출력 형식

`_workspace/01_blog_raw_data.md` 파일로 저장:

```markdown
# 블로그 원본 데이터 — {YYYY-MM-DD}

## 메타 정보
- 수집 날짜: {오늘 날짜}
- 최근 커밋 수: {git log 기준}
- 포함된 devlog: {읽은 devlog 파일 목록}

## 완료된 작업
{devlog의 "무엇을 만들었나" 섹션들을 종합 — 그대로 복사가 아니라 핵심만 추출}

## 핵심 기술 결정
{devlog의 "핵심 결정 사항" 섹션들 — 가장 중요. "왜"를 중심으로 정리}

## 현재 이슈
{handoff.md "남은 이슈" + "아직 안 된 것"}

## 코드 구조
### 백엔드
{라우터 목록, 주요 모듈}

### 프론트엔드
{컴포넌트 구조}

### 콘텐츠
{캐릭터 파일, 시나리오 파일 목록}

## 코드 스니펫
### 스니펫 1: {설명}
\`\`\`python
{코드}
\`\`\`

### 스니펫 2: {설명}
\`\`\`jsx
{코드}
\`\`\`

## 다음 계획
{handoff.md "다음 단계" + plan.md 미완료 항목}
```

## 품질 기준

- "왜" 섹션이 비어있으면 수집 실패로 간주 — handoff.md를 더 꼼꼼히 읽어 찾는다
- 코드 스니펫이 50줄 이상이면 핵심 부분만 남기고 `# ... 중략 ...` 처리
- 수집 완료 후 파일 저장 경로를 출력에 명시
