# 블로그 초안 작성 스킬

`_workspace/01_blog_raw_data.md`를 읽어 개발자 독자를 위한 블로그 포스트 초안을 작성한다.

## 작성 전 확인

1. `_workspace/01_blog_raw_data.md` 파일 읽기
2. "핵심 기술 결정" 섹션 확인 — 이 섹션이 포스트의 핵심이 된다
3. 코드 스니펫 수 확인 — 0개면 텍스트 기반 아키텍처 설명으로 대체

## 포스트 구조 (필수 준수)

### 1. 제목
- 이번 개발의 **결정 사항**을 담는다
- 나쁜 예: "김지윤 키우기 Day 2 개발기"
- 좋은 예: "NPC 1:1 채팅 대신 씬 기반 방식을 선택한 이유"
- 좋은 예: "FastAPI + React로 LLM 게임 API 설계하기"

### 2. TL;DR (3줄 이내)
이 포스트에서 얻어갈 3가지를 bullet point로:
```
- 이번에 만든 것: ...
- 핵심 결정: ...
- 다음 단계: ...
```

### 3. 배경
- 이 작업을 왜 시작했는가
- 이전 상태는 어땠는가 (Day 1이면 "처음부터", Day 2 이상이면 이전 handoff 참조)
- 1~2 단락, 200자 이내

### 4. 무엇을 만들었나
- 구체적 결과물을 bullet list로
- 파일/컴포넌트 이름을 코드 인라인으로 표시 (`backend/main.py`, `ChatScreen.jsx` 등)
- "열심히 만들었다" 같은 추상적 표현 금지

### 5. 핵심 기술 결정 (가장 중요한 섹션)
- 이번 작업에서 가장 중요한 설계/구조 결정 1~2개를 상세히 서술
- 형식: **[결정]**: 왜 이렇게 했는가 → 어떤 효과가 있는가
- 비교표(Before/After) 사용 시 독자 이해도 향상

예시 형식:
```
### NPC 채팅 → 씬 기반 방식 전환

처음에는 NPC별로 채팅방을 만드는 구조였다. 그런데 테스트해보니 플레이어가 
항상 말을 먼저 걸어야 해서 "내가 왜 이 NPC한테 말 걸어야 하지?" 느낌이 들었다.

씬 기반으로 바꾸니 플레이어가 **김지윤의 대사를 입력**하는 방식이 됐다. 
"여기서 김지윤이라면 뭐라고 할까?" 하는 몰입감이 생겼다.
```

### 6. 코드로 보는 구현
- 코드 스니펫 1~2개 + 각 코드에 대한 설명 (2~3문장)
- 코드가 없으면 이 섹션 생략하고 대신 아키텍처 다이어그램(ASCII) 사용

### 7. 아직 남은 과제
- 현재 안 되는 것들을 솔직하게 서술
- "문제가 있다"가 아니라 상태 + 해결 계획을 함께 서술
- 예: "Node.js가 없어 프론트엔드를 아직 실행 못 했다. 다음 단계에서 설치 후 통합 테스트 예정."

### 8. 마무리
- 이번 작업에서 배운 점 1가지
- 다음 포스트 예고 (다음 단계 계획 기반)
- 2~3문장

## 문체 가이드

- 1인칭 서술 권장: "~했다", "~이었다"
- 독자에게 질문 던지기 허용: "여러분은 이런 경우 어떻게 하셨나요?"
- 기술 용어 첫 등장 시 괄호로 설명: `FastAPI (Python 기반 웹 프레임워크)`
- 단락 최대 4줄 — 길면 분리

## 출력 프로토콜

파일: `_workspace/02_blog_draft.md`

파일 상단에 frontmatter 포함:
```yaml
---
title: {제목}
date: {오늘 날짜 YYYY-MM-DD}
tags: [개발일지, 게임개발, FastAPI, React, LLM]
series: 김지윤 키우기 개발기
---
```

완료 후 파일 경로와 포스트 제목을 출력에 명시한다.
